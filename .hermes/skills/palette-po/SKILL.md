# Palette PO — 자연어 task 를 잘 정리된 GitHub issue 로 변환

이 skill 은 사용자의 어수선한 자연어 task 설명을 받아 **palette repo 의 SOT
(GLOSSARY / ARCHITECTURE / CONVENTIONS / 최근 ADR)** 와 대조해 잘 정리된
GitHub issue 를 만들고 `ah:needs-execution` 라벨을 부착한다. 그 후 Hermes cron
의 `palette-pm` 이 5분 안에 픽업 → `palette-executor` 가 PR 생성.

다이어그램 상 위치:
```
human → task queue → [palette-po (이 skill)] → github issue + ah:needs-execution → cron 영역
```

> 동일 instructions 의 **Claude Code slash command** 도 별도 존재:
> `palette/.claude/commands/add-task.md` — Claude Code 채팅에서 `/add-task <설명>` 으로 호출.
> 둘은 같은 단계(다이어그램의 `po skill` 박스) 의 양 트랙.

## Prerequisites

- `gh` CLI 인증 완료 (`gh auth status`) — c-yeonwoo/palette 에 **write 권한 필요**
- 환경변수 `PALETTE_AGENT_PAT` 가 있으면 그것을, 없으면 `gh` 가 인증된 계정 사용
- palette repo 가 `~/dev-private/palette` 에 clone 되어 있어야 SOT 읽기 가능

## Inputs

사용자 자연어 (한국어 OK). 예:
- "친구추가 후 mypage 의 지인 카운트가 +1 안 됨"
- "ProfileEditScreen 의 키 슬라이더 단위 cm 명시 누락"
- "AdminUsers 정렬 옵션 추가 (가입일/마지막활동)"

## Workflow

### 1. SOT 가벼운 스캔

다음 파일들의 헤더만 읽어 컨텍스트 잡기 (full read 금지 — 토큰 절약):
- `~/dev-private/palette/docs/GLOSSARY.md` — 도메인 용어
- `~/dev-private/palette/docs/ARCHITECTURE.md` — 패키지 구조
- `~/dev-private/palette/docs/CONVENTIONS.md` — 절대 룰
- `~/dev-private/palette/docs/DECISIONS/` 의 최근 3개 ADR title

### 2. Scope 판정

| 조건 | 판단 |
|------|------|
| 단일 도메인 (user/profile/matchmaker/friendship/...) 안에서 | OK — 진행 |
| 다중 도메인 OR 파일 10개+ 변경 예상 | ⚠️ 분할 권장 — 사용자에게 "이대로 갈래 vs 쪼갤래" 확인 |
| 도메인 모델 신규 (새 aggregate) 또는 외부 연동 신규 | 🔴 ADR 필요 — issue body 에 명시 |
| 단순 텍스트/오타/copy 정정 | OK — 빠른 진행 |

### 3. Issue body 템플릿

```markdown
## Scope
<한 단락. 무엇을 / 왜 / 어떤 결과>

## Affected files (예상)
- `path/to/file1.kt` (어떤 변경)
- `path/to/file2.tsx` (어떤 변경)

## Acceptance criteria
- [ ] <행동 1 — 검증 가능한 형태>
- [ ] <행동 2>
- [ ] 백엔드 테스트 통과 (`./gradlew test`) — 변경 시
- [ ] 프론트 테스트 통과 (`cd frontend && npx vitest run`) — 변경 시

## Out of scope
- <이 task 에서 안 할 것 — 다음 issue 후보>

## Hints
- 관련 ADR: <docs/DECISIONS/NNNN-...>
- 도메인 용어: GLOSSARY.md 참고
- <기타 executor 한테 도움될 1~2줄>

---
_🤖 Generated by palette-po (Hermes skill)_
```

### 4. Title 도출

- 70 char 이내 한국어
- `[domain] 무엇을 동사형` — 예:
  - `friendship: 친구 수락 후 mypage 지인 카운트 +1 보정`
  - `profile: 키 슬라이더 cm 단위 명시`
  - `admin: AdminUsers 정렬 옵션 추가`

### 5. GitHub issue 생성

```bash
gh issue create \
  -R c-yeonwoo/palette \
  --label "ah:needs-execution" \
  --title "<도출한 title>" \
  --body "$(cat <<'EOF'
<위 템플릿 채운 body>
EOF
)"
```

성공 시 stdout 의 issue URL 캡쳐. 한 줄 보고:
```
✅ #<number> 생성 — <title>
   <url>
   다음 5분 안에 Hermes cron (palette-pm) 이 픽업 → executor 가 PR 생성합니다.
```

### 6. 권한 오류 시 (HTTP 403/404)

`gh CLI` 가 read-only 계정으로 인증된 경우:
- 안내: `export PALETTE_AGENT_PAT=ghp_...` 필요
- 또는 `gh auth switch` 로 write 권한 계정 사용
- Issue body 는 출력만 하고 실제 create 는 보류

## Constraints (절대 금지)

- `gh issue create` 한 줄로 끝내기 — Scope/AC/Hints 없으면 executor 가 시간 낭비
- main 변경 / 코드 직접 수정 — PO 는 issue 만 만듦
- 라벨 `ah:needs-execution` 외 다른 ah:* 라벨 부착 (queue 우회 금지)
- 동일 title 의 open issue 가 이미 있는데 중복 생성 (`gh issue list` 로 확인)
- 너무 큰 scope 라고 판단되면 사용자 확인 없이 임의 분할 (제안만)

## Example

**입력:** `친구추가 후 mypage 의 지인 카운트가 +1 안 됨`

**결과 issue:**

```
title: friendship: 친구 수락 후 MyPage 지인 카운트 즉시 갱신

## Scope
친구 요청 수락 후 MyPage 의 "지인 N명" 카운트가 즉시 +1 되지 않음.
새로고침해야 반영됨. friendship.accept 응답에서 카운트 동기화 필요.

## Affected files (예상)
- `src/main/kotlin/.../presentation/friendship/FriendshipController.kt` (acceptFriendRequest 응답 보강)
- `frontend/src/app/components/MyPageScreen.tsx` (낙관적 업데이트 또는 invalidate)
- `frontend/src/app/components/FriendConnectScreen.tsx` (수락 후 callback)

## Acceptance criteria
- [ ] 친구 요청 수락 시 MyPage 의 지인 카운트 즉시 +1 (새로고침 X)
- [ ] vitest 통과
- [ ] 거절 시 카운트 변화 없음 (회귀 X)

## Out of scope
- 카운트 실시간 동기화 (WebSocket) — 다음 phase
- 친구 삭제 시 -1 — 별도 issue

## Hints
- 관련 ADR: docs/DECISIONS/0016-application-ui-removal-friendship-unification.md
- 도메인: GLOSSARY.md 의 Friendship 흐름
- 백엔드 API: GET /api/v1/friends 로 fresh count 받을 수 있음
```
