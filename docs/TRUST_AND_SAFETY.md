# Palette — Trust & Safety (어뷰징 대응)

> 어뷰징/사기/안전 위협 모델과 대응. **사람 + 워커 공동 관리** (코드 가드 변경은 ADR 동반).
> 관련 결정: `docs/DECISIONS/0022-anti-abuse-baseline.md`

---

## 0. 큰 그림 — 3개의 공격면

1. **지인 네트워크의 진정성** — "믿을 만한 사람이 소개"가 전제. 가짜 그래프(모르는 사람을 1촌으로)가 들어오면 서비스 정체성 붕괴. 최대 공격면.
2. **현금 유출 경로** — 주선자 포인트 → 현금 출금. 직접 손실 벡터.
3. **베타 신원검증 공백** — NICE DEV bypass + 휴대폰 bypass + 신뢰도 mock. *현재 가짜 프로필이 trivial*. 닫힌 초대 베타라 한정되나 **출금·정식 출시 전 반드시 해제 + 가드 선행**.

> 설계 이점: 수익 모델이 **열람권(외부 결제의 %를 커미션)** 이라, 플랫폼이 현금을 직접 지급하던 구조 대비 **셀프매칭 직접 수익이 사라짐**. 직접 수익을 노리면 결제사기(도용카드+차지백) 또는 감사포인트 파밍이 필요 → 이 둘이 최우선.

## 2. 구현 완료 (this PR — ADR 0022)

### 2.1 MatchabilityFilter — 소개/주선 받기 서버 강제
- `ProfileSettings.canReceiveMatches() = isVisible() && isAcceptingMatches` (도메인 단일 기준).
- 강제 지점:
  - `FeedController.getHomeFeed` — 비매칭 대상자 피드에서 제외
  - `AiSignalController` 후보 계산 — 비매칭 제외
  - `MatchmakingController.createMatchmakingRequest` — 비매칭 대상에게 요청 시 403
- 효과: "소개받기/주선받기 off"가 실제로 노출·요청을 막음 (이전엔 DB 저장만, 필터 미사용 — 프라이버시 + 원치않는 노출 어뷰징 해소). 기존 ADR 0005 의 미해결 항목 해소.

### 2.2 출금 본인인증 게이트
- `MatchmakerController.requestWithdrawal` — `isPhoneVerified` 필수(403 아니면 차단).
- 현금 유출 경로의 1차 신원 게이트. 정식: NICE 실인증 + 계좌 실명(CI) 일치.

## 2-1. 추가 구현 (ADR 0023)

### 출금 holding + 한도 + cooldown
- 즉시 출금 → **HOLD 라이프사이클**: 요청 시 pending 예약, holding(14일) 후 자동 확정(scheduler), 의심건은 관리자 거절(예약 해제).
- 신규계정 cooldown(7일), 일(20만)/월(100만) 한도, 최소 5,000P. `GET /me/withdrawals`, `AdminWithdrawalsController`.

### Rate limit
- `RateLimiter`(Redis/in-memory 분기) — 친구 요청 30/일·매칭 요청 10/일·Nudge 30/일 (429).

### 유저간 신고/차단 + 양방향 격리
- `BlockReportController`(block/unblock/list/report), `AdminReportsController`(검토 큐).
- **양방향 격리**: 차단 쌍을 피드·AI시그널·매칭요청에서 모두 제외 (`BlockService`).

## 3. 다음 구현 (우선순위)

### P0 — 출금/정식 전 필수
- **출금 holding period** (예 14일) + 일/월 한도 + 신규계정 cooldown — 새 `WithdrawalRequest` 라이프사이클 엔티티(PENDING→HOLD→PAID/REJECTED).
- **결제 Toss 서버검증** — AI Signal/열람권 결제 `paymentKey` 존재 체크 → confirm/webhook 으로 금액·상태 검증. 차지백 시 커미션 자동 회수.
- **감사포인트 비출금화** — 신설 시 별도 비출금 잔액 또는 `withdrawable=false` 플래그.
- **NICE·휴대폰 실인증 게이팅** — 출금·연락처 교환·주선자 활동을 실인증 완료자 한정.
- **콜루전 기본 탐지** — 디바이스/IP/결제수단/CI 공유, 폐쇄 루프(소수 계정 반복 매칭), 가입↔성사 시간 비정상 → 의심점수 임계 시 출금 보류 + 운영자 리뷰.

### P1
- **Rate limit** — 친구 요청·매칭 요청·Nudge (Redis 토큰버킷, prod Redis 활용).
- **유저간 신고/차단** — `Report`/`Block` 엔티티 + 양방향 격리(피드·매칭·검색 상호 비노출).
- **디바이스 핑거프린트/IP 캡처** — 가입·출금·매칭 등 키 이벤트에 시그널 기록(콜루전 탐지 입력).
- **열람권 서버측 권한** — 상세 조회 시 ticket 보유/차감 서버 검증(클라 우회 불가), 재열람 무료 서버 기록 기준.

### P2
- **Vision/이미지 perceptual hash** — 사진 도용(동일 이미지 타계정/웹 유명사진) 탐지, NSFW, 라이브니스.
- **리뷰 verified-match gate** — 성사 당사자만 리뷰. 리그는 콜루전 필터 통과 성사만 카운트.
- **운영자 어뷰징 대시보드** — 의심 계정/거래 큐, 출금 보류 관리.

## 4. 탐지 규칙 (콜루전 — 설계 메모)

의심점수(0~100) 가중합, 임계 초과 시 출금 자동 보류:
- 동일 디바이스/IP 로 가입한 계정끼리 매칭 (+높음)
- 동일 결제수단/CI 공유 (+치명)
- 폐쇄 루프: 한 군집(≤5 계정)끼리만 반복 매칭, 외부 매칭 0 (+높음)
- 가입 후 첫 성사까지 시간 비정상적으로 짧음 (+중간)
- 역할 중첩: 주선자=요청자=대상자가 소수 계정 안에서 순환 (+높음)
- 성사 직후 계정 비활성/탈퇴 (+중간)

---

## 5. 내장 설계 원칙

1. **돈은 검증된 신원에게만** — 출금 = 본인인증(현 휴대폰, 정식 NICE+계좌 CI)+holding.
2. **현금성 보상은 외부 결제에서만 파생** — 플랫폼 직접 지급 포인트(감사포인트)는 비출금.
3. **신뢰 신호는 서버가 강제** — 토글·쿨타임·열람권·시드격리 전부 서버 진실원천. (쿨타임/시드격리/MatchabilityFilter 이미 서버 강제)
4. **모든 현금 이벤트는 회수 가능** — 차지백/부정 시 커미션 자동 회수 경로 확보.
