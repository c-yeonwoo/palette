# 0071 — 내친소 게시판 (지인 추천형 디스커버리) + WANTED(후속 실험)

- **상태**: Accepted (내친소 = Phase 2 채택) · Proposed (WANTED = Phase 3 실험)
- **작성일**: 2026-06-21
- **작성자**: ys.choi
- **선행**: ADR 0013(주선자 흐름), 0047(팔레트 Pick 매칭), 0064(무현금 보상), 0065(연락처 핸드오프), 0066(채팅), T&S 문서
- **관련 트렌드**: "셀소(셀프 소개팅/소개주선)" 유행 — 단, 팔레트는 **오픈 자기전시(셀소) 대신 신뢰 기반 변주**로 흡수.

---

## 1. 문제 / 의도
주선자가 없거나 지인망이 얇은 사용자는 매칭 풀을 못 받는다(콜드스타트). "셀소 게시판"이 유행이지만, **오픈 자기소개 피드**는 팔레트 정체성을 깬다:
- 탈스펙 브랜드 ↔ 피드는 인기/외모 투표로 회귀
- "1:1 신중한 연결" ↔ "다대다 구경"으로 희석 (리트머스: *"연결하러 vs 구경하러"*)
- 신뢰망 우회 + 박제(스샷 유출)·성비 폭주 등 T&S 비용 급증

검토한 3안 중 판단(스코어카드 근거):
1. **셀소(자기소개 게시판)** — ✕ 정체성·안전 최악(오픈 자기전시). 채택 안 함.
2. **내친소(내 친구를 소개합니다)** — ◎ **채택**. 셀소의 자기PR을 *제3자(주선자) 보증*으로 치환 → 신뢰↑·박제↓, 주선자(공급) 엔진 강화.
3. **WANTED(이런 사람 찾아요 / 역경매)** — ○ 재미·차별 1등이나 신뢰망 우회 + 게시자 권력 집중 + 유동성 의존 → **Phase 3 실험으로 보류**.

## 2. 결정

### 2-1. 내친소 (채택)
주선자가 **자신의 지인 1명을 게시글로 추천**하고, 관심 있는 사용자가 신청 → 주선자가 신청자 중 1명을 선택 → 기존 2단계 승인 매칭으로 연결.

**흐름**
```
① 주선자가 지인 선택 + 추천사 작성 → ② 대상(지인) 동의 요청
   → (동의) 게시글 ACTIVE  → ③ 사용자가 게시판 탐색 → 신청(물감 차감)
   → ④ 주선자가 신청자 프로필 검토 → 1명 선택
   → ⑤ MatchmakingRequest 생성(기존 엔진): 신청자=요청자, 지인=대상, 주선자=주선자
   → ⑥ 2단계 승인(대상 동의 → 최종) → 연락처 핸드오프(ADR 0065) → 성사 시 주선자 커미션(ADR 0064)
```

**정체성·안전 가드 (필수)**
- **대상자 옵트인 필수**: 주선자가 당사자 모르게 박제 불가. 게시는 `PENDING_CONSENT` → 대상 동의해야 `ACTIVE`. 동의 철회 시 즉시 비활성.
- **사진 단계 공개**: 게시글엔 색깔·닉네임·추천사(+선택적 블러 1장)만. 실사진은 *선택/매칭 단계*에서 점진 공개.
- **신청 = 물감**: 무분별 신청 필터 + 진지함↑ (남 프로필 열람 10물감과 정합, 내친소 신청은 더 낮게 책정 검토).
- **차단 격리 재사용**: 차단 관계면 게시/신청 상호 비노출 (양방향 격리 ADR T&S).
- **rate-limit**: 게시 생성·신청 횟수 제한(RateLimiter 재사용).
- **신고 카테고리 추가**: "동의 없는 게시"·"부적절한 추천사" → 어드민 모더레이션 큐.

### 2-2. WANTED (Proposed — Phase 3 실험)
수요자가 *원하는 결*을 게시(자기 전시 X, 탈스펙 정합) → 해당된다 생각하는 사용자가 지원 → 게시자가 1명 선택 → 자기 프로필 전달 → 지원자 최종 결정. 내친소와 **신청·선택·프로필 교환 머신을 공유**하되 `postType=WANTED`(주선자·대상 없음, 게시자=당사자). 유동성·성비·게시자 권력 집중을 a/b로 관측 후 정식화 판단.

### 2-3. 도메인 (재사용 우선)
- 신규: `IntroductionPost`(주선자·대상·추천사·동의상태·status[PENDING_CONSENT/ACTIVE/CLOSED/MATCHED/EXPIRED]·만료일) + `IntroductionApplication`(게시글·신청자·status[APPLIED/SELECTED/DECLINED/WITHDRAWN]).
- 재사용: `MatchmakingRequest`(2단계 승인 엔진) — 선택 시점에 생성해 기존 흐름에 위임. `Friendship`(대상 = 주선자의 지인). `Matchmaker`(커미션/레벨). `BillingService`(물감 차감). T&S(신고·차단·rate-limit·격리).
- `postType` 필드로 내친소/ WANTED 공용 테이블.

## 3. 트레이드오프 / 한계
- **유동성 의존**: 신청자 풀이 얇으면 죽은 게시판. → 출시 전 충분한 활성 사용자/주선자 확보 후. 초기엔 주선자 커뮤니티(앞선 합의)와 함께 공급 측부터.
- **게시판 = 목적지화 위험**: "구경" 유발 방지 — 신청은 물감·신중, 피드는 무한스크롤/좋아요 없이 카드+필터 위주. 컬러 트라이브 같은 잡담은 배제.
- **모더레이션 부담**: 동의 옵트인 + 신고 큐 + rate-limit 로 1차 방어. 운영 인력 필요.
- **WANTED 신뢰망 우회**: 정식 채택 시 팔레트 신뢰 모델과 충돌 소지 → 실험에서 성비·권력집중 데이터로 검증 후 결정.

## 4. 개발 단계 (요약 — 상세는 BACKLOG NC-/WT- 항목)
- **Phase A(백엔드 코어)**: 도메인+마이그레이션 → 동의 플로우 → 게시판 조회 → 신청(물감) → 주선자 선택→MatchmakingRequest 연결.
- **Phase B(프론트)**: 게시판·게시 생성·동의 화면·신청자 관리·진입점(주선자 대시보드+Discovery)+알림.
- **Phase C(안전·정책·계측)**: 신고/모더레이션·POLICY/약관·퍼널 지표.
- **Phase D(후속)**: WANTED(postType 분기) 실험.
