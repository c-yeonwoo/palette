- **상태**: Accepted
- **결정일**: 2026-06-06

## 컨텍스트

사용자 관점에서 비슷하게 들리는 용어가 난립해 혼동을 유발했다:

- **소개받기**(피드 노출 토글) vs **주선받기**(소개 요청 수신 토글) — 둘 다 "받기"라 무엇이 켜/꺼지는지 불명확.
- **주선 요청**(사용자가 누군가와 이어달라고 함) vs **주선자**(이어주는 사람) — 같은 "주선"이라 행위 주체가 섞임.
- **친구 요청 / 지인 등록 요청 / 연결 제안 / 소개** 등이 의미가 겹쳐 보임.
- **2촌 / 3촌** 촌수 표현이 노후돼 서비스 톤과 안 맞음.

근본 문제: "주선"이라는 한 단어가 **주선자의 역할**과 **사용자의 요청 행위** 양쪽에 쓰였다.

## 결정

### 1. "주선" = 주선자 역할어로 한정

| 맥락 | 용어 |
|------|------|
| 주선자의 역할·화면·활동 | **주선** (주선자, 주선 대시보드, 주선 활동, 주선 성사) |
| 사용자가 이어달라 요청 | **소개 요청** (was 주선 요청) |
| 주선자가 먼저 두 지인 제안 | **소개 제안** (was 연결 제안) |

### 2. "소개" 패밀리로 행위 통일 (Layer 2)

- `MatchmakingRequest` → 사용자 문구 **소개 요청**
- `Nudge` → **소개 제안**
- `Friendship`(PENDING) → **지인 요청** (was 친구 요청 / 지인 등록 요청, 1개 어휘로 통일)

### 3. 토글 역할 분리 (Layer 3)

| 토글 | 백엔드 | 의미 | was |
|------|--------|------|-----|
| **프로필 공개** | `visibility`/`hiddenAt` | 지인 피드 노출 여부 | 소개받기 / 피드 노출 |
| **소개 받기** | `isAcceptingMatches` | 소개 요청 수신 여부 | 주선받기 |

→ "소개만 받기"(공개 ON·소개받기 OFF) 같은 의도를 명확히 표현 가능.

### 4. 촌수 표현 폐기

- 2촌 → **친구의 친구**, 3촌 → **한 다리 더 건너**.

### 5. 앱 내 탭/타이틀 정렬

- 홈 피드 타이틀 "주변 지인" → **"소개"**
- 하단 2번째 탭 → **"인연"** (`IntroductionHistoryScreen`, 추후 매칭 완료 시 채팅 수용)

## 적용 범위 (이번 커밋)

- **프론트**: `MainFeedScreen`(타이틀·뱃지), `ProfileDetailScreen`(촌수·소개 요청 문구), `App.tsx`(탭), `MyPageScreen`(토글 라벨·토스트·확인 다이얼로그), `IntroductionHistoryScreen`/`ConnectorDashboard`(소개 요청 문구). "주선 요청" 15곳 → "소개 요청"; "주선을 요청" → "소개를 요청".
