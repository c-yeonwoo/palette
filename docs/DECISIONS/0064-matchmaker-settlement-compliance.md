# 0064 — 주선자 정산 컴플라이언스 & 무현금 보상 모델 검토

- **상태**: Proposed (의사결정 대기 — 변호사/세무사/금융규제 자문 필요)
- **작성일**: 2026-06-14
- **작성자**: ys.choi

## 1. 문제

주선자(개인 지인)에게 **실제 현금**을 정산/지급하는 흐름이 미해결. 현재 [POLICY §2.6](../POLICY.md) 에 "물감 → 본인 명의 계좌 출금(3영업일)" 모델이 문서화돼 있으나, 이 흐름은 아래 3개 규제 레이어를 건드린다.

### 레이어 1 — 세무/소득 (무조건 해당)
- 개인에게 보상 지급 = 사업소득(프리랜서 3.3% 원천징수) 또는 기타소득. 플랫폼은 **원천징수 + 지급명세서 제출** 의무. 주선자는 일정액 초과 시 종합소득세 신고 대상.

### 레이어 2 — 전자금융거래법
- 사용자 결제금을 모아 주선자에게 분배 = 지급대행/자금이체 성격 → 전금법 등록 이슈(PG·선불전자지급수단).
- **"물감"이 선불전자지급수단으로 분류될 소지**. 폐쇄형(자사 서비스 내 사용 + 환불 제한)이면 면제 범위가 넓지만, **현금 출금을 붙이는 순간** 성격이 바뀐다.

### 레이어 3 — 결혼중개업 + 다단계 (벤더로 해결 불가, 최대 리스크)
- **결혼중개업법**([§7](../POLICY.md)): 영리 만남 알선 + 주선자 수수료 → 무신고 중개업 소지.
- **다단계(방문판매법)** — 현재 설계가 *주선자 레벨제 + 친구 초대 보너스 + 레벨별 분배(15~40%)*. "사람을 데려오고 그 사람의 활동으로 보상받는" 구조는 후원방문판매/다단계로 해석될 수 있고, 공정위 미등록 시 형사처벌 대상. **현금 보상이 붙을수록 위험이 커진다.**

## 2. 후보 모델

| 모델 | 주선자 보상 | 레이어1(세무) | 레이어2(전금법) | 레이어3(중개·다단계) | 주선자 유인 |
|---|---|---|---|---|---|
| **A. 현금 정산** (현행 지향) | 물감 → 현금 출금 | 원천징수·지급명세서 필수 | 등록 PG/정산대행 필요 + 물감 선불수단 검토 | 높음(현금보상+레벨=다단계 시그널 강함) | 강 (현금) |
| **B. 폐쇄형 무현금 보상** | 본인 데이팅에 쓰는 물감·열람권·구독 크레딧 (출금 X) | 거의 해당 없음(현금 미지급) | 폐쇄형 선불 → 면제 범위 넓음 | 크게 완화(회사가 현금 안 줌) | 중 (서비스 가치) |
| **C. P2P 자발 성의표시(QR)** | 매칭된 사용자가 주선자에게 직접(앱 외부 계좌). 앱 수익 0 | 플랫폼엔 무(個인 간). 주선자 본인 소득세는 별개 | 앱이 자금 미보유·미이체 | 완화되나 "앱이 알선+보수 구조화" 논쟁 여지 | 중 (현금이지만 비보장) |

> B와 C는 **결합 가능** (폐쇄형 크레딧 + 선택적 P2P 성의표시).

## 3. "앱→주선자 현금 정산 제거" 시 구조·정체성 변화 (사용자 요청 검토)

핵심 질문: *플랫폼이 주선자에게 현금을 지급하는 프로세스를 아예 없애면?*

### 3.1 규제 — 대폭 경감
- **레이어 1(세무) 소멸**: 플랫폼이 주선자에게 지급하는 돈이 없으니 원천징수/지급명세서 의무가 사라짐.
- **레이어 2(전금법) 경감**: 물감을 *사용자용 폐쇄형 재화*(열람·언락·구독)로만 두면 선불수단 면제 범위. 출금/이체 기능 자체가 없어 자금이체업 이슈 해소.
- **레이어 3(다단계) 급감**: 다단계의 본질은 "회사가 주는 후원수당". **회사 보상이 0이면 다단계 구성요건이 대부분 붕괴.** (단, 레벨/초대 보상을 *물감*으로도 주면 잔여 논쟁 가능 → 직접 주선 성과 위주로 단순화 권장.)
- **결혼중개업**: 여전히 "영리 만남 알선" 논점은 남지만, 수익이 *주선 수수료*가 아니라 *사용자 구독/언락*에서 나오면 일반 데이팅 앱에 가깝게 포지셔닝됨.

### 3.2 수익 구조 — 전환
- 현행 가정: 사용자 결제 → 일부를 주선자에게 분배(플랫폼이 중간). 
- 전환 후: **수익은 100% 사용자 측**(소개요청 티켓·열람·팔레트 Pick 구독). 주선자에게 나가는 현금 0 → 분배 회계·정산 인프라 전부 불필요. 마진 단순/개선.

### 3.3 정체성 — "돈 버는 주선" → "신뢰 기반 호의 주선"
- 기존 피칭: "주선자 인센티브 시스템(레벨별 30~50% 커미션)" = 부업/수익형 주선자.
- 전환 후: 주선자는 **지인을 위한 호의 + 사회적 자본(명예·뱃지·성사 스토리) + 본인 데이팅 혜택**으로 움직임. 지인 네트워크 기반("신뢰가 연결을 만든다")이라는 원래 정체성과 오히려 더 정합적. 단, "용돈벌이" 기대로 모은 초기 주선자 풀의 동기는 약화 → 초기 공급(주선자 수) 확보 전략을 다시 설계해야 함.

### 3.4 비용 — 트레이드오프
- 잃는 것: 강한 현금 유인. 부업형 헤비 주선자 이탈 가능.
- 얻는 것: 규제 3개 레이어 대부분 회피, 정산 인프라/리스크/세무 부담 제거, 출시 속도, 어뷰징(현금 출금 farming) 표면적 축소.

## 4. 무현금 주선자 유인 툴박스 (현금 없이 활동 동기)

1. **본인 데이팅 혜택(가장 강력·합법적)**: 주선 성공 시 *본인이 쓰는* 물감/무료 열람권/구독 크레딧 적립. 폐쇄형이라 전금법·세무 트리거 없음. "남을 이어주면 내 만남이 쉬워진다."
2. **사회적 자본**: 주선자 레벨·뱃지·명예의 전당·"이 커플을 이어준 사람" 크레딧·성사 스토리. (현행 레벨 시스템에서 *돈만 떼고 명예는 유지*.)
3. **관계적 보람**: 친구 둘을 행복하게 했다는 만족 + 후기/감사 메시지 노출.
4. **게이미피케이션**: 시즌 랭킹, 연속 주선, 한정 뱃지, 미션.
5. **P2P 자발 성의표시(QR)** — §5 참고. 앱은 비중개·수익 0.

## 5. P2P 성의표시(QR) — ADR 0046 과의 충돌 (반드시 인지)

사용자 아이디어: *매칭 성공 시 주선자 계좌 QR을 띄워 매칭된 사람이 직접 성의표시, 앱은 수수료 포기.*

- **이것은 [ADR 0046](0046-external-payment-abuse-guard.md) / [POLICY §6](../POLICY.md)(외부 송금 유도 금지)를 정면으로 뒤집는다.** §6은 90/10 인앱 분배를 보호하려 외부송금을 *금지*했는데, 무현금 모델로 가면 인앱 분배 자체가 없어지므로 §6의 전제가 사라진다. → 모델 B/C 채택 시 ADR 0046·§6 **개정 또는 폐기**가 동반돼야 함.
- 남는 리스크(전환 시 재검토 필요):
  - **세무**: 주선자가 P2P로 반복 수령 시 본인이 소득 신고 의무(플랫폼 무관하나, 안내 책임/평판 이슈).
  - **어뷰징**: 가짜 매칭으로 성의표시 farming, 앱 밖 사기·환불 불가, 검증 불가.
  - **중개 보수 구조화 논쟁**: 앱이 QR을 *띄우고 유도*하면 "사실상 알선 보수 중개"로 볼 여지 → "전적으로 자발적 사례, 앱은 일절 관여·수취 안 함" 문구/UX 가 필수.
- 대안 절충: QR 대신 **앱 내 무현금 '감사 보내기'**(물감·배지·감사카드)만 허용하면 외부 현금 논쟁을 피하면서 보상 표현은 유지.

## 7. 외부 자문 질의서 초안

**변호사 (규제·소비자)**
1. 주선자에게 (a)현금 (b)폐쇄형 포인트만 지급할 때 각각 결혼중개업법·방문판매법(다단계)·전자금융거래법 적용 여부?
2. 레벨제 + 친구 초대 보상 + 성사 분배 구조가 다단계판매/후원방문판매에 해당하나? 해당을 피하려면 어떤 설계 제약이 필요한가?
3. 앱이 주선자 계좌 QR을 노출해 사용자→주선자 P2P 송금을 *유도*하면 알선 보수 중개로 볼 소지가 있나?
4. "결혼" 비표방 + 지인 추천 데이팅의 결혼중개업 신고 필요성?

**세무사**
5. 플랫폼이 개인 주선자에게 지급 시 원천징수율·지급명세서·연 한도? 사업소득(3.3%) vs 기타소득 구분 기준?
6. P2P(앱 비관여) 수령 시 주선자 개인의 신고 의무와 플랫폼의 안내 책임 범위?

**금융규제(전금법)**
7. "물감" 폐쇄형 포인트의 선불전자지급수단 등록 면제 요건? 현금 출금 추가 시 등록 의무 발생 임계?
8. 직접 라이선스 없이 등록 PG/정산대행을 끼는 표준 구조와 책임 분담?

## 8. 권고 (자문 전 잠정)

- **베타 = 모델 B(폐쇄형 무현금)로 운영.** 출금 끄고 주선 보상을 *본인 데이팅 혜택 + 명예*로. 규제 3레이어를 대부분 회피하며 "주선이 실제로 작동하는가"를 검증.
- **정식 전환 시** 자문 결과로 A 복귀(정산 인프라 구축) 또는 B/C 확정. A로 가면 §1의 정산대행·세무 스택 + 다단계 재설계가 전제.
- 코드 영향: 현재 `WithdrawalRequest`/출금 흐름은 **기능 플래그로 비활성**해두고 도메인은 보존(전환 대비). 신규 작업 아님 — 결정 후 별도 ADR.
