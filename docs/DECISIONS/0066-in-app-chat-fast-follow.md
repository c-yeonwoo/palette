# 0066 — 인앱 채팅 fast-follow (핸드오프 안전 레이어)

- **상태**: Proposed (권고안 포함 — 베타 핸드오프 운영 데이터 확인 후 착수 결정)
- **작성일**: 2026-06-20
- **작성자**: ys.choi
- **선행**: ADR 0065(매칭 성사 연락처 핸드오프), ADR 0051(관계 단계·종결), 기존 `UserReport`/`UserBlock`(유저 신고·차단), [POLICY §6 외부 송금 금지](../POLICY.md), 계정 삭제 의무(ADR P0)

---

## 1. 문제

ADR 0065 로 매칭 성사 핸드오프는 **정직해졌지만**(상대 카카오톡 ID/전화 노출), 외부 채널로 대화가 빠지면서 두 가지 약점이 남았다.

1. **안전망 이탈** — 대화가 카카오톡/문자로 넘어가면 팔레트의 신고·차단·증거 보존이 닿지 않는다. 분쟁·괴롭힘·외부 송금 사기(§6) 발생 시 플랫폼이 개입할 근거가 사라진다.
2. **즉시 전화번호 노출 마찰** — 성사 즉시 전화번호를 교환해야 첫 연락이 가능. "아직 번호까지는…" 단계의 사용자에게 프라이버시 부담. 한 쪽이 부담스러우면 핸드오프 자체가 멈춘다.

**인앱 채팅**은 이 둘을 해소하는 안전 레이어다 — 번호 노출 없이 대화를 시작하고, 신뢰가 생기면 그때 ADR 0065 연락처 카드로 외부 채널 전환. 모든 대화가 신고·차단·moderation 범위 안.

## 2. 결정 (권고)

매칭 성사(`COMPLETED`) 관계에 **인앱 1:1 채팅**을 fast-follow 로 추가한다. 자체 채팅은 ADR 0065 연락처 카드를 **대체하지 않고 보완**한다 — 채팅이 안전한 기본값, 연락처 교환은 사용자가 원할 때.

### 2.1 전송 방식 — 폴링 (WebSocket 아님)

베타 규모(CBT 소수 동시 사용자)에서는 **short-polling 을 권고**:
- 스레드 열린 동안 `GET .../messages?after={lastId}` 를 N초(예: 4~6s) 간격 폴링. 백그라운드는 알림(FCM, 기존 PushNotificationService)으로 대체.
- 기존 스택(JWT REST + Spring MVC + H2/MySQL) 그대로 — sticky session·연결 관리·스케일아웃 부담 없음.
- 기존 mock `chat/ChatThread.tsx`(현 `VIEWER_ID="me-001"`)를 폴링 REST 에 배선만 하면 됨.
- **WebSocket/SSE 는 동시성이 정당화될 때 후속 최적화** — 베타에서 선제 도입은 과설계.

### 2.2 잠금·게이팅

- 채팅은 **`COMPLETED` 관계의 두 당사자에게만** (ADR 0065 의 403 당사자 가드 패턴 재사용).
- `ENDED`(만남 피드백 NOT_FOR_ME, ADR 0051) 관계는 **읽기 전용 동결** — 신규 메시지 차단, 기록은 보존(분쟁 대비).
- 전화번호 교환은 **선택**으로 후퇴 — "인앱 채팅"이 1차 핸드오프, "연락처 카드"는 사용자가 누를 때.

### 2.3 보존·삭제

- 메시지는 관계 수명 동안 보존, **`ENDED`/차단 후 90일까지** 분쟁·어뷰징 검토용으로 유지 후 purge.
- **계정 삭제 시 즉시 purge**(계정 삭제 의무) — 상대방 측 스레드에서도 발신 내용 익명화/삭제.

### 2.4 moderation·안전

- 기존 **`UserBlock`** 재사용: 차단 시 양방향 스레드 숨김 + 신규 전송 차단.
- 기존 **`UserReport`** 재사용: 스레드/메시지 신고 → 운영 큐(AdminReportsScreen). §6 외부 송금 유도 신고 카테고리 활용.
- (후속) 발신 시 PII/계좌번호·외부 송금 패턴 경고 배너 — ADR 0046 어뷰징 가드와 연계.

### 2.5 데이터 모델·엔드포인트 (착수 시 설계 확정)

- `chat_messages`: `id`, `request_id`(=관계), `sender_id`, `body`, `created_at`, `read_at?`. 신규 컬럼이므로 prod 마이그레이션은 idempotent `CREATE TABLE` + `SchemaDriftTest` 등록.
- `GET /api/v1/relationships/{requestId}/messages?after={id}` (폴링), `POST .../messages` (전송) — 둘 다 당사자 403 가드 + `COMPLETED` 체크.
- 프론트: `ChatThread` 실배선(mock 제거), "인연" 화면 `MATCHED+` 카드에 채팅 진입 + 안 읽음 배지.

## 3. 트레이드오프

- **폴링 비용**: 열린 스레드당 N초 폴은 소규모 베타에서 무시할 수준. 사용자·동시성 증가 시 SSE/WebSocket 으로 전환(2.1).
- **자체 채팅 ≠ 메신저**: 카카오톡 수준 기능(사진·읽음·이모지)을 기대하면 빈약. 의도적 — "안전한 첫 대화 → 외부 전환"이 목적이지 풀 메신저가 아님.
- **moderation 부하**: 신고 큐·보존이 운영 리소스를 요구. 베타 규모에선 수용 가능, 성장 시 자동 필터 필요(2.4 후속).

## 4. 영향 / 후속

- 본 ADR 은 **설계·결정 문서** — 구현은 BACKLOG `채팅 (CH)` 스프린트(P1).
- ADR 0065 의 死 코드 보존분(`MatchDetailScreen`/`chat/*` mock)이 여기서 재활용/정리됨.
- 착수 트리거 권고: 베타에서 ADR 0065 핸드오프의 실제 마찰(번호 교환 거부율·외부 채널 분쟁 신고)이 관측되면 우선순위 상향.
